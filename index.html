Harness

The question changed: not which model, but which harness

Last year the industry argued about which model. This year nobody argued about models — the whole conference argued about the harness: how you plan, loop, verify, and govern the agents that models power.
And the sharpest debate underneath it — lights-off software factories vs. humans conducting orchestras — landed on the orchestra: a human orchestrating and reviewing the agents, never stepping out of the loop.
Loops

Loops that survive real codebases

  • Kyle Mistele (HumanLayer): blind “run it in a bash loop” agents don’t fit teams with customers and SLAs. Build agentic control loops from control theory instead: set point → sensor (linters, ast-grep) → controller picks one small change → actuator (agent + skill).
  • One iteration per scheduled run, producing one labeled PR — the loop halts while the last PR sits unreviewed. Steering lives in a version-controlled feedback file.
  • Lance Martin: long-horizon reliability is architecture — decouple the harness (“brain”) from sandboxed execution (“hands”); credentials live in a vault, never in the sandbox. Independent verifier loops grade the outcome.
  • Theo Browne: today’s frontier models already orchestrate — spawning sub-agents, splitting and verifying work — on prompt alone. “We need to go bigger.”
The strongest production loops already look like control systems, not chat sessions — a one-open-PR gate and a version-controlled feedback file are the upgrade worth stealing.
Harness

The durable product is the harness around the model

Traces, evals, diagnosis, bounded auto-fix, and a closed improvement loop convert real failures into reusable system knowledge.
  • Observe: traces preserve the context, path, tool calls, cost, latency, and outcome.
  • Diagnose: failures are classified as model, context, tool, policy, orchestration, or UX defects.
  • Codify: the fix becomes an eval, deterministic check, skill, policy, or runtime change.
  • Verify: the new contract is tested before the loop is allowed to expand autonomy.
Value: usage makes the platform more reliable instead of generating the same incident at higher volume. Failure mode: without this loop, more autonomy only accelerates repeat mistakes.

Model routing

The right model is the smallest one that meets the quality bar

Model choice and serving architecture work together. A smaller, task-fit model on an inference-optimized platform can improve latency, cost, and throughput without sacrificing the outcome.
  • Start smaller: route routine, bounded work to smaller models; reserve frontier models for ambiguity, complex reasoning, and higher-risk decisions.
  • Optimize the serving layer: inference-focused platforms such as Together AI improve performance through optimized batching, caching, quantization, and hardware.
  • Measure the real workload: compare task success, p50/p95 latency, and cost per successful task under realistic concurrency — not a single-request benchmark.
Decision: use the smallest model that clears the evals on an optimized serving platform. Add model capability or infrastructure control only when quality, privacy, or the SLO requires it.
Voice

Voice is the stress test for the entire agent architecture

Voice is not an audio wrapper around chat. It is a realtime agent runtime that exposes every weakness in state, latency, tools, and recovery.
  • Conversation mechanics: turn detection, interruption, silence, noise, and full-duplex behavior decide whether the system feels human or broken.
  • Runtime state: policy, context, tools, UI state, and long-running tasks must continue changing while someone speaks.
  • Latency is a budget: speech recognition, end-of-turn detection, model work, tools, and speech generation all spend from the same experience budget.
  • Correctness is observable behavior: did the tool complete, did the user hear the right status, and did the system escalate cleanly?
Required evals: interruption recovery, end-of-turn to first audio, tool completion, state consistency, and escalation — at p50 and tail latency.
Voice

There is no single voice stack

The architecture determines what we can optimize, inspect, and control. The right choice depends on the workflow’s risk — not which demo sounds most human.
ArchitectureStrengthTradeoff
Cascaded
speech → text → reasoning → speech		Modular, inspectable, and easier to connect to tools, policy, and existing text agentsEvery stage adds latency; transcription can discard tone, timing, and ambiguity
Speech-to-speechLower-latency, expressive interaction with more natural timing and prosodyHarder to inspect, govern, ground in tools, and explain when behavior fails
Hybrid production stackSpeech-native interaction with explicit state, tools, policy checks, and verificationMore orchestration, but control and naturalness can be tuned independently
Decision rule: use the simplest architecture that meets both the interaction SLO and the workflow’s control requirements.
Voice

Humanlike behavior is a system metric, not a model benchmark

Users experience one delay, but the budget is spent across end-of-turn detection, recognition, reasoning, tools, speech generation, and the network.
  • Optimize the whole turn: making speech generation faster does not help when tool execution or end-of-turn detection dominates the wait.
  • Measure the tails: averages hide the pauses that make a conversation feel broken. Track p50 and p95 from end-of-user-turn to first meaningful audio.
  • Protect the interaction: false cutoffs, missed barge-in, talking over the user, filler loops, and failure to resume after interruption are product defects.
  • Test conversations, not components: a good ASR score, model score, and TTS score can still combine into a bad call.
Minimum scorecard: end-of-turn to first audio, interruption recovery, false-cutoff rate, task completion, state consistency, escalation quality, and cost per completed outcome.

Google Hardware

Gemma 4 puts frontier techniques on your phone

Google is now shipping models that run on a slice of the phone’s own processor and do the job of far larger, machine-hosted models. Gemma 4 is the clearest proof yet: a noticeably faster generation that trades some latency for running fully on-device, while staying on par with Gemini 3.1 on streaming, audio transcription, and overall intelligence.
  • Advanced reasoning: configurable thinking modes (chain-of-thought inference) built for complex logic and multi-step planning.
  • Multi-token prediction (MTP): built-in drafter models predict several future tokens at once, accelerating local inference with no loss in output quality.
  • Native multimodality: text and images natively; the smaller E2B and E4B variants add native audio for speech recognition and translation; the 12B variant runs visual and audio data directly through the language model, encoder-free.
  • Built for agents: an intelligence-per-parameter design built natively for multi-step reasoning, multimodal processing, and function calling.
The booth’s best moment: this model running live, on a phone, in the demo — AI genuinely at your fingertips.
Hardware Open weights

GLM 5.2 on DGX Station: frontier intelligence under your desk

Frontier-scale intelligence is migrating from the datacenter to the desk, as GPUs and models both get more efficient.
  • Live demo: GLM models running on an NVIDIA DGX Station — Ahmad M. Osman, Founder/CEO, Osmanatic.
  • Near-term roadmap (?): GLM-class models on a single RTX 5090 (64GB) by mid–2027, with a further jump by late 2027 — the exact sequencing wasn’t fully clear from the session.
  • The longer arc: as models get more parameter-efficient, that same capability trickles down to older, lower-tier cards — RTX 30-series through 50-series — not just this year’s flagship.
Hardware efficiency, not just model size, is what determines when frontier intelligence fits under your desk.
Local LLMs Hardware

Local LLMs: hardware is the new oil

NVIDIA and a panel of CEOs made the case that local LLMs are a bigger platform shift than most roadmaps assume — and that hardware, not the model, is the bottleneck to watch.
  • Lowering the bar to run local: vendors are building the tooling that lets developers configure and install local models easily, delivering high performance on far less compute.
  • NVIDIA as the platform: partnered across the ecosystem and positioning itself as the hardware layer underneath both hosting and training.
  • The resource case for local: pitched as a way to cut cost across tokens, hardware, and natural resources like water — not just a latency or privacy win.
Local inference is being pitched as a sustainability story as much as a cost story — a real counterweight to cloud-only strategy.
Model routing

Give each agent the model built for its job

Stop sending every task to one model. Route by task, and let each sub-agent run the model that’s actually strong in that domain.
  • Domain-tuned sub-agents: one model fine-tuned toward science, another toward coding — the platform spins up whichever sub-agent carries the model the task actually needs.
  • Not every agent needs the frontier model: a reporting agent has no use for the top-of-line model when a much smaller, cheaper one does the job just as well.
  • Model routing, defined: given a task, the platform selects the right model for it automatically — no manual wiring required.
Every agent spun up this way carries its own model, context, memory, and tools: a mini harness in its own right.
Model routing

Context lives with one model — cost still has to be earned

Split work across a main model and smaller task models, and one question dominates: where does the context actually live?
  • One model owns the context: the full context stays with a single main model; smaller models get only the slice their task needs, never the whole session.
  • Cheaper per token ≠ cheaper overall: a lower per-token price can still lose to a pricier model once the extra tokens and round-trips it takes to get there are counted — an optimization problem, not a pricing one.
  • The KV cache is the real lever, and it’s still unsolved: context switching runs through it, but there’s no clean way yet to target or optimize work against it directly. (A tool referenced as “Cachette” came up here (?).)
  • New operating habits for multi-model fleets: heartbeat every model, and keep a fallback configured for when one fails.
Harness

From instructions, to agents, to agents on the fly

The harness pattern has moved through four stages, each one stripping out a layer of manual setup.
StageWhat happens
Direct instructionYou tell Claude or Codex what to do, and it does it — no persistent agent structure.
AgentsYou create an agent and assign it a task, a goal, and a role.
Sub-agentsSub-agents get created as needed, per request, under a parent agent.
Agents on the flyYou just give Claude context; it spins up the agents it needs on the fly, using the harness already in place.
Context, memory, and session state across this divide-and-conquer mix of models is still the open problem — the thing every vendor in the room was actively trying to solve.
Browser agents

Browser agents: a small-world moment at the Browserbase booth

Browserbase builds browser-based agents — and it’s already the tool my own team runs on a live RealPage project. Met their sales engineer at the booth: small world.
  • Already in production here: Browserbase is running today on one of my team’s RealPage projects.
  • Prior art: I built a browser agent at a previous company that pulled tax data from county websites and ran OCR on it — the same shape of problem RealPage is solving now.
  • LLMs → LAMs: the same trajectory that turned a chat model into a working tax-scraping OCR agent is now a supported category — browser-based language action models (LAMs) — instead of something every team builds from scratch.
